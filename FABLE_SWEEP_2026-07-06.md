# gp-core Fable Sweep 2026-07-06

Shared-spine audit of the gp-core library that both RigWork and Riffwork depend on. The headline is the entitlement layer: three separate paths (`isPro`, cap helpers, `resolveAccountState`) all grant Pro too generously. An expired 14-day trial resolves to lifetime unlimited Pro, a lapsed "active" subscription stays Pro forever, and `isExpired()` ships correct but is never called. The moment tiers go live, every stale localStorage license is a free Pro key in both apps. Second theme is data integrity: the hand-entered 96-device CC map has real defects (Korg presets with unsendable Program Change numbers) that the current type-only tests structurally cannot catch. Third is quiet contract drift: `normalizeGearItem` does not establish the invariants `createGearItem` promises, the schema says `defKey` while RigWork stores `defId`, and the base64url codec is copy-pasted four times with divergent failure modes. Fix the entitlement gate first, it is the only critical.

## Correctness

**Expired trial keeps full Pro entitlement forever** `[critical/small]`
`src/account/license.js:38` - `isPro()` is `isActive() && !!tier`, and `isActive()` treats status `trialing` as active with no expiry check. `createTrial()` (schema.js:31) sets tier=monthly + status=trialing and is app-managed in localStorage, so nothing server-side ever flips it. Verified by execution: a trial created 20 days ago returns `{ isPro: true, isTrial: true, daysLeft: 0, midiCap: null, songCap: null }` (unlimited). Riffwork already wires `_accountState.isPro` into gating (auth.js:194,377), so at tier launch every expired trial is a lifetime Pro license in both apps.
Fix: make expiry part of the shared gate. `isActive(license, nowMs)` returns false for trialing licenses whose `trial_ends_at <= nowMs`; thread `nowMs` through `isPro`/`isProFor`/`midiDeviceCap`/`cloudSongCap`/`resolveAccountState` (which already receives `nowMs` but never uses it for the pro check). Test: expired trial => isPro false, midiCap 2, songCap 5.

**isExpired() exists but nothing calls it, lapsed "active" subscriptions stay Pro** `[high/small]`
`src/account/session.js:52` - `resolveAccountState()`/`isPro()` never consult `isExpired()`. Verified: a license with status `active` and `current_period_end` 30 days in the past returns `isPro: true` while `isExpired()` returns true. Pro gating depends entirely on the billing webhook flipping `status`; any cached or offline license row grants Pro indefinitely after the sub lapses. The library ships the correct check and ignores it.
Fix: in `resolveAccountState` (or inside `isPro` with `nowMs`), treat active-but-expired subscription tiers as not-Pro, with an optional grace window. Perpetual licenses are already safe (no `current_period_end`). Add tests for lapsed active rows.

**interpolateRigTrack: ramp-over-ramp snaps value back to last settled (or 0)** `[high/small]`
`src/song/rig-track.js:70` - `fromValue = settled.get(key) ?? 0` uses only the last settled value as the ramp start. A new ramp on a (deviceId, cc) while a previous ramp on the same key is still in progress discards the in-flight value. Verified: cue@bar1 ramps CC7 0->100 over 8 bars, cue@bar5 ramps to 20 over 4 bars; at bar 4.9 value is 49, at bar 5.0 it snaps to 0, then ramps 0->20. On stage that is an audible pop on whatever is ramping (volume swell, mix, feedback). Only ramp-onto-ramp is broken.
Fix: when starting a new ramp, resolve the current value at `cue.bar` first. If an in-progress ramp exists for the key, compute its interpolated value at `cue.bar`, use that as `from`, and settle it. One branch; add a ramp-chaining test.

## API & consistency

**normalizeGearItem does not establish the invariants createGearItem guarantees** `[medium/small]`
`src/gear/schema.js:85` - `createGearItem` guarantees `boardIds: []`, `boardId`, `customParams: []`, `presets: []`, but `normalizeGearItem` (which consumers call on every loaded item) only stamps `_v` and short-circuits if `_v` already exists. A legacy or hand-synced item normalizes with `customParams`/`presets`/`boardIds` undefined; any consumer doing `item.presets.map` or `item.boardIds.includes` crashes. RigWork already papers over it with `(dev.customParams || [])` (main.js:95), re-implementing the defaults the schema was meant to own. Related drift: schema documents `defKey`/`devKey` but RigWork stores `defId` (40 uses of `defId` vs 5 of `defKey`/`devKey`).
Fix: make `normalizeGearItem` backfill the same defaults as `createGearItem` (arrays + boardId derivation) even when `_v` is present, and reconcile `defId` vs `devKey`/`defKey` (accept-and-migrate `defId`, or fix the schema doc) before Riffwork starts reading gear items.

**Free-tier caps hardcoded in two places and product-agnostic** `[medium/trivial]`
`src/account/session.js:18` - `FREE_STATE` hardcodes `midiCap: 2, songCap: 5` while `midiDeviceCap`/`cloudSongCap` hardcode 2 and 5 again in license.js:47/55. No shared constants, so a pricing change edited in one file silently diverges between the logged-out path (FREE_STATE) and logged-in path (resolveAccountState). The cap helpers also accept any license without a product check: `midiDeviceCap(activeRiffworkLicense)` returns null (unlimited RigWork devices). `resolveAccountState` pre-filters by product so its path is safe, but the exported helpers are footguns.
Fix: export `FREE_CAPS = { midi: 2, songs: 5 }` (or per-product LIMITS) from license.js and use it in both files; either make the cap helpers take `(license, product)` and check `isProFor`, or document them as internal to `resolveAccountState`.

**resolveAccountState reports tierLabel "Pro (monthly)" on non-Pro past_due/cancelled licenses** `[medium/trivial]`
`src/account/session.js:59` - `tierLabel` is derived from `license.tier` alone. Verified: a past_due monthly license resolves to `{ isPro: false, midiCap: 2, tierLabel: 'Pro (monthly)' }`. Riffwork renders `_accountState.tierLabel` directly (auth.js:371-375), so a lapsed user sees "Pro (monthly)" while every gate treats them as free: a guaranteed confused-user ticket. Also `sessionToLicense`'s priority sort has no tiebreak, so a user holding both a perpetual and an active monthly row for the same product gets whichever the DB returned first (nondeterministic tierLabel/version-lock).
Fix: gate the label: `tierLabel = pro ? getTierLabel(license.tier) : 'Free'` (optionally "Pro (monthly), payment issue" for past_due), and add a tier-rank tiebreak (perpetual > annual > monthly) to `sessionToLicense`'s sort.

## Test gaps

**No data-integrity tests over the 96-device CC map, the critical shared path is shape-checked only** `[medium/small]`
`test/gear.test.js:23` - 7 tests, all asserting field types (cc is a number, label is a string). Nothing validates the invariants the 3545-line hand-entered device-defs data depends on: no duplicate CC within a device, cc/def/preset values within 0-127, options[] bands monotonically increasing and covering to 127, starterPreset ccValues referencing declared params, recallPC within programSelect bounds and <= 127. The file header says "Do not invent values, these fire at real pedals," yet a 30-line mechanical validator found the Korg recallPC 128-199 defect immediately.
Fix: add an invariant test iterating `deviceDefs`: unique CCs per device, all cc/option/preset values 0-127, option `val <= max` strictly increasing ending at 127, preset ccValues keys a subset of param CCs, `recallPC === -1` or within `[programSelect.min, min(programSelect.max, 127)]`. Turns every future data PR into a validated one, including community suggest-gear submissions.

## Move-to-core candidates

**Korg starter presets have recallPC 128-199, unencodable in MIDI, will never recall** `[high/medium]`
`src/gear/device-defs.js:1436` - `korg_minilogue_xd` ships ~72 starterPresets with recallPC 128-199 ("SpiralNebula" PC 128 through "TPL Hats" PC 199), and both Korgs declare `programSelect { min:0, max:199 }`. A MIDI Program Change data byte is 7-bit (0-127): values 128+ cannot be sent, and programs above the first bank need Bank Select (CC0/CC32) + PC, which neither the starterPresets shape nor programSelect can express. A scan of all 96 devices confirms these are the only out-of-range PCs. RigWork indexes `starterPresets` directly (rig/src/main.js:2855), so tapping any of these sends garbage or nothing.
Fix: add optional `bank { msb, lsb }` to starterPresets and programSelect in the shared shape, encode the Korg banks correctly (xd: LSB = floor(program/100), PC = program % 100 per the Korg MIDI implementation), and have RigWork emit Bank Select before PC when present. At minimum, clamp/flag `recallPC > 127` so presets are not silently broken.

**base64url codec implemented three times with divergent failure modes** `[low/small]`
`src/song/showfile.js:188` - the identical JSON -> UTF-8 -> base64url encode/decode is copy-pasted in part/codec.js:10-34, song/rig-track.js:120-142, and song/showfile.js:188-211 (plus a fourth variant in riffwork/src/main.js:7767). Failure modes already differ (decodePart -> null, decodeRigTrack -> [], decodeShowFile -> null), and all three use the O(n^2) `binary += String.fromCharCode(b)` loop, which matters for multi-song Show Files in a URL hash. Any future change (the codec.js header already plans gzip) must be made in three places or share links fork between apps.
Fix: extract `encodeB64url(obj)`/`decodeB64url(str)` into a shared `src/codec.js` used by all callers; build the binary string via chunked `String.fromCharCode.apply` or `TextDecoder('latin1')`. Keep the per-caller fallback values at the call sites.

**normalize* fast-path skips nested normalization and returns the caller's mutable reference** `[low/small]`
`src/song/showfile.js:169` - `normalizeShowFile`/`normalizeSong`/`normalizeShowFileSongEntry` all do `if (raw._v === 1) return raw`. A decoded payload (decodeShowFile accepts arbitrary base64 from a URL hash) carrying top-level `_v:1` bypasses all nested normalization: songs map entries, sections, setlist, rig_track cues are trusted as-is, so a hand-edited or older-app `.gpshow.json` flows un-normalized into both consumers. It also returns the same object reference while `normalizeLicense`/`normalizePart` return copies: inconsistent aliasing across the same library.
Fix: on the `_v:1` fast path, still walk nested collections (cheap: map entries through the entry-level normalizers, which themselves fast-path) and return a shallow copy, matching `normalizeLicense`/`normalizePart`. Document one rule in SCHEMA_RULES.md: normalizers always return a fresh top-level object with normalized children.
